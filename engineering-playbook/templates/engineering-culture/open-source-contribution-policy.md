# Open Source Contribution Policy

*We build our company on open-source software, and we believe in giving back. This policy defines how engineers can contribute to OSS on company time.*

## 1. Core Philosophy
*   **20% Time:** Engineers are encouraged to spend up to 20% of their sprint time contributing to the open-source projects that our company relies on (e.g., submitting a bug fix to the React repository or a Go logging library).
*   **Company Sponsorship:** We actively sponsor projects that are critical to our infrastructure.

## 2. Approved Contributions
*You do not need permission to contribute to OSS if it meets these criteria:*
1.  **Bug Fixes:** Fixing a bug in a library we actively use in production.
2.  **Documentation:** Improving the documentation of a tool we rely on.
3.  **Minor Features:** Adding a minor feature to a library that unblocks our own internal development.

## 3. Restricted Contributions (Requires Approval)
*You MUST get approval from the Chief Architect before:*
1.  **Open-Sourcing Internal Code:** Before extracting an internal company library and publishing it publicly on GitHub, it must undergo a strict security and IP review.
2.  **Major Architectural Changes:** Committing to maintain a massive new feature in an external OSS project requires capacity planning.

## 4. The Golden Rules of OSS Representation
*When you contribute using your `@dependly.com` email address, you represent the company.*
*   Be exceptionally polite and professional in PR comments and GitHub Issues.
*   Do not argue with OSS maintainers.
*   Never accidentally commit proprietary company secrets, API keys, or business logic to a public repository.
